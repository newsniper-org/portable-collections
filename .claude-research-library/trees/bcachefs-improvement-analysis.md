## Formal verification of the multicopy node + recovery layer  — `partial`

**bcachefs today.** A bcachefs btree node is literally a multicopy search structure. Per the doc (1.2, 8.7): a node = one fully-sorted bset (from the last compaction) + appended bsets in journal order; a lookup MERGES across all bsets and must return the newest version of a key; deletes are out-of-place whiteouts (KEY_TYPE_deleted) appended and dropped only at compaction; in-memory bsets are periodically resorted into a geometric progression (<=3 bsets) and fully mergesorted on split/rewrite; keys are packed per-bset via bkey_format. The durability/recovery side: btree updates are journalled (8.6) with monotonic, never-reused sequence numbers; journal_replay re-inserts keys "idempotent - replaying the same entry twice is safe" (8.6.5); ordering across a crash is enforced by sequence blacklisting (1.3, 8.6.5) of bsets whose journal seq is newer than the last durable journal entry, plus a 64-seq padding for in-flight writes. Correctness today rests on tests + ~48 recovery passes with a dependency graph + online fsck (1.3.1), NOT on proofs. Idempotency/restartability is the stated reliability mechanism ("inherently resilient to interruption at any point - including during recovery itself", p8).

**Gap claimed.** The two safety properties that matter most have no machine-checked proof, only test + repair coverage: (1) the in-node MULTICOPY merge/compaction invariant - that merging across bsets always yields search recency (newest version wins), that whiteout-drop at compaction can never resurrect a deleted key, that geometric resort and the per-bset bkey_format pack/unpack roundtrip preserve the logical key->value map; and (2) journal-replay correctness - that replay reconstructs exactly the committed upsert history, that idempotence holds under arbitrary mid-replay crash + restart, and that sequence blacklisting (with the 64-pad and GC-once-unreferenced) drops exactly the non-durable bsets and never a durable one. These are precisely the bugs that "tests + fsck" catch late and probabilistically, and they are the same one-vs-other-copy desync class this workspace exists to make unrepresentable - here the two "copies" are bsets of one key, and the desync is a stale/resurrected version surviving compaction or replay.

**Proposal (from Verifying Concurrent Multicopy Search Structures (Patel, Krishna, Shasha, Wies 2021), Verifying Concurrent Search Structure Templates (Krishna, Patel, Shasha, Wies, PLDI 2020), Verifying Lock-free Search Structure Templates (Patel, Shasha, Wies 2024)).** Instantiate the Patel-Krishna-Shasha-Wies multicopy-search-structure (MCS) template against the btree-node-as-bset-list, modelling: each bset = an MCS node in the in-node DAG; the bset's journal sequence number = the MCS upsert timestamp t (the doc already has a total monotonic seq, so HUnique/HClock fall out directly); whiteouts = MCS tombstones; bset compaction/geometric-resort = MCS compaction (keep newest copy, discard tombstones); a node lookup = MCS search, whose obligation is SEARCH RECENCY. Prove once that the node-level operations (append-bset upsert, merge-on-read search, resort/compaction) satisfy the MCS template's node specs, which discharges within-node linearizability by construction. Then verify the bkey_format pack/unpack as the MCS node-level representation refinement (val(C_n) is preserved). For the durability layer, model journal replay as replaying the MCS upsert history H: prove replay reconstructs H-up-to-last_seq and is idempotent under restart (this maps to MCS's "upsert history only grows" monotone-ghost-state argument), and prove sequence blacklisting computes exactly the durable prefix of H. Use the CSS-templates flow framework (edgesets + flow interface) for the orthogonal btree TOPOLOGY proof (parent/child key-range coverage = check_topology), keeping node-internal MCS reasoning decoupled from tree-shape reasoning. Use the 2024 hindsight/lock-free formalization for the six-locks optimistic-relock linearization points (seqlock sequence numbers = future-dependent LP, handled by hindsight rather than per-site prophecy). Scope realistically: do the node+replay MCS proof in Iris/Coq as a model (a la the paper's verified in-memory LSM), not a C-extraction; treat it as a verified reference model that the C is differentially tested against.

**Already in bcachefs?** partially: bcachefs has the structural ingredients the proofs assume - a monotone never-reused journal sequence number (= MCS timestamp), idempotent/restartable ops, tombstone-style whiteouts, compaction-keeps-newest, and a heavily engineered recovery+online-fsck regime that empirically enforces these invariants. What is absent is any MACHINE-CHECKED proof: correctness is tests + ~48 recovery passes + fsck, explicitly not proofs. So the design is "proof-ready" in shape but unverified; the multicopy-node and journal-replay invariants in particular have never been formally connected to the MCS/CSS template results despite being a near-textbook instance of them.

**Skeptic's strongest counterargument.** bcachefs has deliberately chosen repair-first assurance over proof-first, and the doc states the rationale explicitly: because every op is idempotent and restartable, "the filesystem is inherently resilient to interruption at any point - including during recovery itself" (p8), and after unclean shutdown "no extra repair passes are needed because the filesystem is always consistent after journal replay" (p9). The proposal targets exactly the layer that is already nearly trivially correct: re-inserting a key is an idempotent upsert into a sorted structure, and within-node "newest-bset-wins" merge recency is local and well-bounded. Meanwhile the doc's own ~30 check_* passes show that the real operational corruption surface is the CROSS-btree referential invariants (check_backpointers_to_extents, check_alloc_info, check_extents, nlink, directory/subvolume structure, snapshot consistency) — none of which a node-level MCS proof touches. So the "highest-blast-radius invariant" claim is inverted: MCS would verify the least failure-prone layer while leaving the actual field-corruption sources to fsck. Snapshots compound this: snapshot-ID-in-bpos is a second multicopy axis threaded through extents/inodes/dirents/xattrs with its own delete_dead_snapshots / reconstruct_snapshots / interior-collapse machinery; a node-merge proof that abstracts it away is proving a model that doesn't match production semantics. And the proposal itself concedes it verifies a model, not the unsafe C, so fsck must remain regardless — meaning the deliverable is a differential-testing oracle, not a correctness guarantee, competing with online-fsck which already serves as a far-broader runtime oracle. The proof is orthogonal to (not aligned with) every constraint bcachefs actually fights: bounded latency, an evolving on-disk format, and the model-to-C gap.

**What survives → recommendation.** Drop the broad "verify the multicopy node" framing — within-node merge recency and journal-replay idempotency are too local/obvious to justify Iris/Coq cost, and the snapshot axis makes a faithful node model intractable. What genuinely remains worth doing is the ONE narrow, self-contained subproperty the proposal buries: the sequence-blacklisting durable-prefix argument. Prove (even in a lightweight model checker / TLA+ rather than Iris) that the blacklist computation — last_seq window + 64-seq in-flight padding + "blacklist GC only once no on-disk btree node references the sequence" + never-reused 64-bit counter — drops EXACTLY the non-durable bsets and never a durable one, under arbitrary crash interleavings including crash-during-recovery. This is a small total-order property, it is genuinely under-covered by tests (the crash-interleaving + GC-timing space is combinatorial), and a bug there is silent and catastrophic (resurrected stale btree node or dropped committed update). That is a tractable, high-value target that respects the constraints. Everything else in the proposal (full MCS template instantiation, CSS flow-framework topology proof, hindsight six-locks linearization) is academically elegant but buys assurance bcachefs already gets more cheaply from idempotency + online fsck.

## Zoned/flash-native allocation to kill the double-GC  — `partial`

**bcachefs today.** bcachefs's allocator is already a host-side FTL: §1.4 divides each device into buckets (512K-2M), writes each bucket sequentially/append-only, never overwrites until the whole bucket is invalidated and its generation number is bumped, and runs copygc to evacuate mostly-empty buckets (8% default reserve, 5-21% configurable; a fragmentation-LRU btree picks targets). §8.2 makes the lifecycle explicit: buckets cycle free -> dirty/cached -> need_discard -> free, the background allocator runs invalidate/discard(TRIM)/freelist pipelines, and write points segregate IO by inode-hash and data type (user/btree/copygc/reconcile) so data with similar lifetime shares a bucket and whole buckets free at once. State lives in alloc + freespace + need_discard + bucket_gens + LRU btrees plus a write-buffered backpointers btree (physical->logical). Crucially the doc ALREADY names the target: §1.4 says copygc+allocator are 'essentially doing the same things as the flash translation layer in SSDs', and that 'the same algorithms are intended to be used for managing SMR and zoned hard drives directly... Buckets map naturally to zones: allocate a bucket, write append-only, never write again until we discard and reuse the whole thing - exactly the semantics a zoned device requires.'

**Gap claimed.** Three concrete gaps remain between the doc's stated 'buckets map naturally to zones' aspiration and a true ZNS-native data path. (1) On a CONVENTIONAL flash SSD the doc admits the double-GC tax persists: bcachefs copygc moves live data to defragment buckets, then the SSD's own FTL independently GCs erase blocks and reserves 7-28% over-provisioning (ZNS paper Fig.1/§2.1) - two independent copying GCs stacked, each paying write-amp and each reserving capacity. The doc only says buckets map naturally to zones; it does not describe the host-side machinery a real ZNS target needs. (2) The bucket model has no notion of the ZNS active/open-zone limit (8-32 zones, ZNS §3.1) - bcachefs's write points are unbounded in principle and bucket sizes (down to 512K) are far smaller than ZNS zone capacities (hundreds of MB to low GB), so a naive bucket==zone mapping either wastes the open-zone budget or forces multi-bucket zones with no enforced sequential-write ordering or whole-zone reset semantics. (3) The bucket_gens btree + key cache front a generation-stale-check and hot-key path, but there is no explicit demand-paged, locality-aware residency policy for the alloc/backpointer working set the way DFTL caches only the hot slice of the page map in SRAM; the LRU btree handles cached-data eviction, not metadata-map residency under memory pressure.

**Proposal (from ZNS: Avoiding the Block Interface Tax for Flash-based SSDs (Bjorling et al., USENIX ATC 2021), F2FS: A New File System for Flash Storage (Lee et al., USENIX FAST 2015), DFTL: A Flash Translation Layer Employing Demand-based Selective Caching of Page-level Address Mappings (Gupta, Kim, Urgaonkar, ASPLOS 2009)).** Add a true ZNS/zoned data path as an f2fs-style mapping, keeping the existing bucket abstraction as the host-FTL it already is, and overlay a DFTL-style demand-cache on the alloc-key hot path. Concretely: (a) Make zone the allocation domain and bucket the sub-unit, exactly f2fs's section=consecutive-segments / zone=series-of-sections hierarchy: a bcachefs 'zone' = N contiguous buckets sized to the device's reported zone capacity, with a per-zone write pointer and a whole-zone reset that maps onto the ZNS zone-reset / TRIM-the-whole-zone command. This is a small extension to the alloc btree (a zone-id + write-pointer field) reusing the existing need_discard -> free transition as zone-reset. (b) Bound concurrent open zones to the device's active-zone limit by capping the number of simultaneously-open write points to the ZNS open-zone budget, and route bcachefs's existing data-type write points (user/btree/copygc/reconcile + inode-hash) into f2fs multi-head-logging hot/warm/cold streams so lifetime-correlated data co-locates in a zone and whole zones free at once - this is bcachefs's own 'segregate by write point' insight made zone-aligned, and it directly attacks copygc work. (c) When the backend is ZNS, DISABLE copygc's defrag role inside a zone (a zone is reset whole, never partially compacted) and let copygc operate only at zone granularity for cold-zone consolidation, so the host stops duplicating the FTL's intra-erase-block GC; the device's in-place GC and over-provisioning vanish (ZNS §3.1). (d) Adopt f2fs's greedy + cost-benefit victim selection (already mirrored by bcachefs's fragmentation-LRU btree) for ZONE selection. (e) For the metadata hot path, add a DFTL-style demand-paged cache (CMT-equivalent) for alloc keys / backpointers keyed on temporal locality, so under memory pressure only the hot working set of the bucket/backpointer map stays resident while the cold map lives in its btree (which it already does) - sizing the residency by access recency rather than pinning, the way DFTL keeps only the hot page-map slice in SRAM.

**Already in bcachefs?** partially: the doc explicitly states the intent ('buckets map naturally to zones... exactly the semantics a zoned device requires', §1.4) and the append-only/gen-bump/whole-bucket-reuse model plus need_discard->free+TRIM lifecycle and lifetime-segregating write points are already the right primitives - bcachefs is genuinely a host-side FTL by design. What is NOT in the doc: (a) a zone=N-buckets hierarchy with a per-zone write pointer and whole-zone reset, (b) any handling of the ZNS active/open-zone limit and its interaction with watermarks, (c) f2fs-style multi-head hot/warm/cold zone packing tied to the active-zone budget, (d) copygc demoted to zone-granularity on ZNS to stop duplicating intra-zone FTL GC, and (e) a DFTL-style demand-paged residency policy for the alloc/backpointer map (the LRU btree governs cached-DATA eviction, not metadata-map residency). So the foundation is present and intentional; the concrete ZNS data path and the demand-cache are the additions.

**Skeptic's strongest counterargument.** The proposal splits into two halves that fare very differently against the doc. (1) The ZNS data path is genuinely not-built and the open-zone-limit-vs-watermark interaction is real and hard — but the doc shows it is SMALLER and MORE-AUTOMATIC than framed. §8.2.1 (p66) says buckets are "typically 512 KB to 16 MB" and auto-sized, not "down to 512K far smaller than zones," so "zone = N contiguous buckets with a write pointer" is close to a sizing/grouping change plus a WP field, not a new on-disk regime. copygc "demotion to zone granularity" (proposal c) is largely a CONSEQUENCE, not new machinery: §8.2.8.2 shows bucket state is derived, and the existing write-point segregation (§8.2.2.1: user/btree/copygc/reconcile + per-inode-hash, "whole buckets become free at once") already means a whole-zone-reset device has no intra-zone fragmented targets for copygc to chase. Worse for the proposal, that very inode-hash segregation is UNBOUNDED in write points, which directly COLLIDES with a 8-32 open-zone budget — a tension the proposal names but understates (it's not just reserving zones for low watermarks per §8.2.4; the whole segregation strategy may not fit the open-zone budget). (2) The DFTL demand-cache (proposal e) is the weak half and is largely ALREADY-DONE: the doc already provides demand-paged, locality-aware metadata residency. The btree node cache is a hash table keyed by physical pointer with reclaim (a demand-paged residency policy); §8.2.8.5 bucket_gens packs 256 gens/key SPECIFICALLY so "the RCU read path reads a small bucket-gens key rather than the full alloc key"; the key cache fronts hot single-key btrees; the write buffer batches backpointer/LRU/accounting. So "keep only the hot working-set of the bucket/backpointer map resident, cold map in its btree" is what node-cache + bucket_gens + key-cache + write-buffer already deliver. A separate DFTL CMT layer would DUPLICATE the node cache and violate the everything-is-one-btree uniformity, and its demand-miss adds a btree lookup on exactly the hot path bucket_gens was built to keep cheap (the proposal admits the latency risk but misses that bucket_gens already solved it). The proposal's own "already_in_bcachefs" concedes the LRU btree governs cached-DATA eviction not metadata residency — but it misattributes: metadata residency is the NODE CACHE's job, and that is already demand-paged.

**What survives → recommendation.** Drop the DFTL/CMT demand-cache (e) entirely — it duplicates the node cache + key cache + write buffer and reinvents the bucket_gens btree (§8.2.8.5), and it adds an on-the-hot-path btree-miss latency that violates the soft-realtime goal to solve a problem already solved. Keep only the genuinely-novel ZNS half, and narrow it to its load-bearing piece: the open/active-zone budget as a FIRST-CLASS scarce resource that the watermark system (§8.2.4) must reserve against. Concretely: (i) add a per-zone write-pointer field to the alloc/freespace btrees and a whole-zone-reset that reuses the need_discard->free transition (this part the proposal gets right and it stays an alloc-btree attribute, preserving uniformity); (ii) treat OPEN ZONES, not free buckets, as the reserved resource — guarantee a reserved open zone for reclaim/btree-split/interior_updates so writeback can never deadlock, AND cap or coalesce the per-inode-hash + data-type write points so their count fits the device active-zone limit (this collapse of unbounded write points into a bounded open-zone budget is the real design work, and the f2fs hot/warm/cold multi-head logging is the right model for it); (iii) for crash-consistency, recover the host write pointer by reading the DEVICE's report-zones on mount and cross-checking against the alloc btree in the existing §8.2.10.1 runtime-check path — cheaper and more robust than the proposal's "recoverable via journal replay," and it closes the host-WP-vs-device-WP divergence corruption class. The benefit (kill the in-device GC + over-provisioning on ZNS) is real and the foundation/intent is documented (§1.4: "buckets map naturally to zones... exactly the semantics a zoned device requires"), so this is a credible engineering project — but its size is ~the f2fs ZNS delta (~hundreds of LoC) for the data path, with the open-zone/watermark interaction being the only deep part, NOT a metadata-caching overhaul.

## Copygc batch sizing & old-node reclamation  — `partial`

**bcachefs today.** bcachefs separates the two "free" surfaces. (1) ON-DISK: copygc uses a fragmentation-LRU btree to find the most-fragmented buckets, relocates their live data via backpointers, and frees whole buckets; freed buckets transition dirty/cached -> need_discard -> free, and a separate discard worker batch-iterates the need_discard btree to send TRIM (1.4 p13; 8.1.10 p65; 8.2.3 p68; 8.2.8.4 need-discard btree p73). copygc_reserve (8%, configurable 5-21%) and a tiered watermark stack (normal/copygc/btree/btree_copygc/reclaim/interior_updates, 8.2.4 p68) guarantee forward progress under pressure; runtime checks rate-limit recovery passes. (2) IN-MEMORY: the doc states struct btree node objects are essentially never freed during normal operation -- the node cache is a hash table keyed by the physical on-disk pointer and CoW splits allocate new on-disk nodes while reusing pooled in-core objects; roots are pinned. Concurrency is six-locks + seqlock-like optimistic relocking + database-style deadlock cycle detection + restart, with NO IO under btree locks.

**Gap claimed.** The "Are Your Epochs Too Epic" RBF anti-pattern (a single large free-batch overflowing an allocator thread-cache and triggering a tens-of-ms flush, stalling all threads -- Kim/Brown/Singh 2024) is mostly DESIGNED OUT for the hot path because btree objects are pooled, not freed. But three batch surfaces remain that the doc does NOT show as amortized: (a) the node-cache SHRINKER -- under kernel memory pressure the cache must actually free a batch of struct btree objects (and their large 128-256K buffers) back to the kernel allocator; a reclaim-driven shrink that walks and frees a large LRU run is precisely a synchronous large-batch free, and freeing a 256K node buffer is a high-latency operation that can run under reclaim with soft-realtime expectations. (b) The DISCARD worker drains the entire need_discard btree as a batch; a burst of bucket invalidations (e.g. a large truncate/snapshot-delete) produces a TRIM storm with no per-tick budget described, and TRIM latency on real SSDs is highly variable. (c) The copygc reserve is a STATIC percentage (8%), not adaptive to fragmentation rate or device free-bandwidth; the epoch paper's core lesson is that the right knob is bounding work-per-event, not pre-reserving capacity. None of (a)-(c) is shown to use the amortized-free / bounded-reservation discipline the papers advocate.

**Proposal (from Are Your Epochs Too Epic? Batch Free Can Be Harmful (Kim, Brown, Singh, PPoPP 2024) -- RBF problem + amortized-free fix, Reclaiming Memory for Lock-Free Data Structures: There has to be a Better Way (Brown, 2017) -- DEBRA/DEBRA+ signal-based bounded O(mn^2) reservation; fault tolerance to stalled threads, Interval-Based Memory Reclamation (Wen, Izraelevitz, Cai, Beadle, Scott, PPoPP 2018) -- bounding memory a stalled-but-not-crashed thread can pin).** Apply the "amortized free" (AF) discipline from Kim/Brown/Singh and the bounded-reservation idea from IBR to the three batch surfaces, with a single shared per-tick byte/op budget: (1) Node-cache shrinker: instead of freeing N nodes synchronously when the shrinker is invoked, move evicted nodes to a bounded auxiliary "to-free" list and release a small fixed quantum per shrinker tick / per allocation slow-path call (the AF "free gradually over time, creating reuse opportunities" technique), so a starving reclaim thread never serializes a large 256K-buffer free run; cap the to-free list length (IBR-style bounded reservation) so a stalled reclaimer can hold back only O(budget) memory, not unboundedly. (2) Discard worker: give the need_discard drain a per-iteration TRIM budget (bytes or count) measured against observed TRIM completion latency, spreading a bucket-invalidation burst across ticks rather than one storm -- exactly the "amortize the batch over time" fix. (3) Make copygc_reserve adaptive: keep the 8% floor for safety but let an EWMA of fragmentation-creation rate vs copygc throughput raise/lower an effective working reserve, and bound copygc's per-invocation evacuation to a quantum so a single copygc pass can't monopolize the device write path. Crucially, key the budgets off Linux's existing reclaim/latency signals (the "signaling" angle of Brown 2017 and the OS-quiescent-state angle of Hart 2006) rather than a hardcoded constant.

**Already in bcachefs?** partially: bcachefs already has the two strongest structural defenses against the epoch paper's anti-pattern -- (i) btree node objects are pooled and essentially never freed on the hot path, so the RBF "large batch overwhelms the thread-cache" failure largely cannot occur there, and (ii) recovery passes are explicitly rate-limited and the watermark stack guarantees forward progress. What is NOT shown in the doc, and is the genuine gap, is per-tick BOUNDED-BUDGET amortization on the three edges that do free in bulk: the node-cache shrinker's synchronous large-buffer free, the discard worker's whole-need_discard-btree TRIM drain, and copygc's static (non-adaptive, non-quantized) reserve/evacuation. So: the philosophy is present (reserves, rate limiting, pooling) but the specific amortized-free / bounded-reservation discipline on those bulk-free edges is at best implicit and not documented.

**Skeptic's strongest counterargument.** Two of the three "un-amortized" surfaces are mischaracterized by the proposal, and the third is the only real gap. (1) The DISCARD worker is already an amortized pipeline, not a batch flush. Section 8.2.3 lists Discard as one of three continuous pipelines of the *background allocator* ("runs continuously, producing free buckets"), and 8.2.8.4 says the discard worker *iterates* the need_discard btree and clears the flag per-key via a trigger. A btree-iteration background worker is structurally an amortized drain with natural backpressure (it competes for the same six-locks/transaction budget as everything else and holds NO btree lock across the TRIM IO, per the hard constraint), and on no-discard devices it is a no-op. The proposal's premise of a synchronous "TRIM storm with no per-tick budget" is not what the doc describes; adding an explicit byte/op budget is a tuning knob on an already-continuous worker, not a structural fix. (2) The COPYGC reserve being "static 8%" is a deliberate, documented safety floor, and copygc is already "automatic and continuous" (8.1.10) reading the fragmentation-LRU btree incrementally rather than evacuating in one monopolizing pass; copygc also runs under its own write point (8.2.2.1) and its own watermark tier (copygc/btree_copygc, 8.2.4) precisely so it cannot starve user writes. The 5-21% configurability already exposes the space/safety tradeoff to the operator. Making the reserve EWMA-adaptive adds a control loop that directly fights the doc's single most-emphasized invariant — the tiered watermark forward-progress guarantee that "critical internal operations can always make progress, even when the filesystem is full" — and risks the exact near-full panic class bcachefs is built to avoid (drop the effective reserve via a mistuned EWMA right as fragmentation spikes and copygc has nowhere to relocate). Overstreet clearly already considered near-full behavior: the whole watermark stack and the "~90% capacity" guidance exist for it. (3) The genuinely unaddressed surface is the node-cache SHRINKER: the doc states struct btree objects are "essentially never freed during normal operation," which means the doc simply does not cover the kernel-memory-pressure path where the shrinker MUST return 128-256K node buffers to the kernel allocator — and that synchronous large-buffer free run under reclaim is exactly where the epoch-paper mechanism (bounded per-tick free quantum + capped to-free list) transfers, and exactly where the proposal itself admits the paper transfers "least cleanly."

**What survives → recommendation.** Drop surfaces (b) discard and (c) adaptive reserve — both are either already amortized (discard is a continuous background pipeline with built-in transaction/lock backpressure and no-IO-under-locks) or deliberately rejected (the static reserve + tiered watermarks are the documented forward-progress guarantee, and an EWMA reserve is a control loop that endangers it; if anything, a per-invocation copygc evacuation quantum may already be implicit in its incremental fragmentation-LRU scan). Keep ONLY surface (a), narrowed: the btree node-cache shrinker under kernel memory pressure. The doc's "objects essentially never freed" covers steady state but is silent on reclaim, so there is a real, undocumented edge where a large synchronous free of 256K buffers can run with soft-realtime expectations. The defensible, kernel-idiomatic change is to make the shrinker's `scan_objects` return work in bounded quanta (which the Linux shrinker API already supports via nr_to_scan and SHRINK_STOP) and move evicted nodes to a length-capped to-free list released gradually on the allocation slow path, giving an IBR-style bound on how much a stalled reclaimer pins. Crucially this must (i) never free a pinned/root/intent-locked node, (ii) never hold a btree lock across the free, and (iii) be validated to not let the cache grow unbounded under sustained pressure (the to-free cap + reuse-on-slow-path is what prevents that). This is a localized latency fix to one path, not the three-surface shared-budget control system originally proposed.

## Snapshot deletion/GC & in-key snapshot IDs  — `partial`

**bcachefs today.** bcachefs encodes the snapshot ID directly in every key's bpos (inode, offset, snapshot) and stores all versions of a key in the SAME snapshot-aware btree (only extents/inodes/dirents/xattrs are snapshot-aware). Creation is genuinely O(1): "two new snapshot nodes are allocated as children... No keys are copied" (8.3.4), so it is NOT btrfs-style COW-btree cloning. The price is paid at deletion: 8.3.5 states deletion cost "is proportional to the volume of data in the deleted snapshot, since the thread must walk every snapshot-aware btree to remove or relocate affected keys." It is two-phase: a background thread (auto_snapshot_deletion) deletes/whiteouts leaf-snapshot keys and moves keys of single-surviving-child interior nodes; interior-node depth/skiplist updates are deferred to single-threaded recovery (8.3.5). It is self-healing/restartable and batches multiple deletions into one pass; a delete_dead_snapshots recovery pass (8.3.7) catches leftovers. Reads walk UP the snapshot tree using a per-node 128-bit ancestor bitmap (O(1) within 128 IDs) + a randomized 3-entry skiplist (O(log n) deeper) to test "is X an ancestor of Y" (8.3.3).

**Gap claimed.** Two concrete weaknesses. (1) DELETE COST IS UNBOUNDED BY THE DEAD DATA but effectively bounded by a FULL SCAN of all snapshot-aware btrees: there is no per-snapshot index of "which keys carry this snapshot ID," so even deleting a snapshot that mutated 3 keys forces the thread to traverse the snapshot-bearing portions of 4 btrees to find them (whiteouts and shared keys have no reverse map). This is the mirror image of btrfs's problem (btrfs is O(1)-delete-marker but O(data) at create; bcachefs is O(1)-create but O(scan) at delete). (2) RANGE SCANS ACROSS A VERSION pay an ancestor-resolution probe PER KEY: every lookup in a snapshot must, for each candidate key in the range, test ancestorship and check that "no closer ancestor has overwritten it" (8.3.3) — versions of unrelated snapshots are physically interleaved in bpos order within the same bset, so a scan in one snapshot reads and discards keys belonging to siblings. Deep or wide snapshot trees inflate both the bytes scanned and the per-key ancestor work.

**Proposal (from B-trees, Shadowing, and Clones (Rodeh 2008) — lazy free-space refcounts; delete visits only nodes belonging solely to the deleted tree (5.4); clone/delete as refcount inc/dec, How to Copy Files (Zhan et al., FAST 2020) — copy-on-abundant-write: defer materializing a private copy until accrued divergence justifies the rewrite, decoupling copy granularity from write granularity, Persistent Non-Blocking BSTs Supporting Wait-Free Range Queries (Fatourou & Ruppert 2018) — version-stamped persistent nodes + sequence-number phasing so a range scan sees a consistent version without per-key re-resolution; informs (D) below).** Add a snapshot-scoped reverse index plus selective subtree-sharing, keeping the existing in-key snapshot model as the default. Concretely, three layered changes: (A) A new "snapshot_keys" write-buffer-backed btree keyed (snapshot_id, btree_id, bpos) that records, at write time, every key that was *originated in* (tagged with) a given snapshot ID — i.e. a per-snapshot dirty-set. This is exactly the bcachefs write-buffer pattern (high-churn, coalescing, periodic flush) already used for backpointers/LRU/accounting, so it reuses existing infra and is crash-consistent via the journal. Deletion then becomes: range-scan snapshot_keys for the dead ID(s) and touch only those keys — turning O(scan all snapshot btrees) into O(keys actually owned by the dead snapshot + their whiteout fixups), mirroring Rodeh's principle that delete should visit only nodes that "belong only to" the deleted tree (Rodeh 5.4 step 2). (B) For interior/shared structure, adopt Rodeh-style lazy refcounts at the granularity of the snapshot TREE's btree subtrees that are provably single-snapshot (a subtree all of whose keys carry one snapshot ID and that snapshot's whole lineage is dead): such a subtree can be freed by a single refcount-drop-to-zero (Rodeh 5.4 step 1) instead of key-by-key whiteout walking — recovering btrfs/WAFL's cheap whole-subtree reclamation for the common "delete a leaf snapshot that diverged a lot" case, while keeping O(1) create. (C) Borrow "How to Copy Files" copy-on-abundant-write as the divergence policy for WHEN to split a shared subtree into a private one: keep snapshots sharing physical keys until accrued divergence in a subtree crosses a threshold, then materialize a private subtree — so the snapshot_keys reverse index and the refcounted-subtree fast path stay small and deletion stays cheap, instead of every tiny edit fragmenting sharing.

**Already in bcachefs?** partially: bcachefs ALREADY made the hard call that distinguishes it from btrfs/Rodeh/WAFL — O(1) create via in-key snapshot IDs and shared keys, with a self-healing, batched, restartable background+recovery deletion pass and O(1) ancestor bitmaps (8.3.3-8.3.5). It does NOT have a per-snapshot reverse index (deletion is explicitly a full scan of snapshot-aware btrees, 8.3.5), nor refcounted subtree sharing (sharing is at the key level, not the subtree level), nor a copy-on-abundant-write divergence threshold (writes diverge per-key immediately). The write-buffer + accounting infrastructure needed to build (A) cheaply already exists. So: the cheap-create half is solved and arguably better than the cited papers; the cheap-DELETE half (reverse index / refcounted subtree reclamation) is the genuine open gap the doc itself names.

**Skeptic's strongest counterargument.** The doc's own wording undercuts the premise of part (A). Section 8.3.5 says deletion cost is "proportional to the volume of data in the deleted snapshot" — Overstreet's claim is that the thread relocates/whiteouts the keys that BELONG to the dead snapshot, not that cost scales with total filesystem size. The proposal restates this as "effectively bounded by a FULL SCAN of all snapshot-aware btrees," but that conflates "must iterate the snapshot-bearing keyspace to FIND the dead keys" with "must rewrite O(total data)." The real cost is a sequential, sorted, prefetch-friendly btree iteration (eytzinger search trees, packed bkeys, write-buffer coalescing) over four btrees whose snapshot-aware portion is dense and cache-resident — i.e. a scan bcachefs is already extremely good at, restartable and self-healing. A snapshot_keys reverse index (B) keyed (snapshot_id, btree_id, bpos) collides head-on with the single most load-bearing invariant in the system: bcachefs deliberately put the snapshot ID IN THE KEY precisely so there is NO separate per-snapshot structure to keep in sync — that is the whole soundness story (the same "make desync unrepresentable" reasoning the user's own BTreeBimap embodies). Adding a reverse index recreates exactly the two-structure-must-agree desync hazard, now needing a new check_snapshot_keys cross-check wired into the dependency-ordered recovery passes (check_snapshots/check_subvols/delete_dead_snapshots) AND online fsck. Worse, part (B)'s refcounted single-snapshot subtrees structurally REVERTS bcachefs to the btrfs/WAFL/Rodeh model of coupling logical sharing to physical node lifetime — the exact coupling 8.3.4 cites as the thing that makes btrfs create O(data) and that bcachefs escaped. The proposal admits this is "a genuine philosophical regression," and under the uniformity constraint (one mechanism for all 28 btrees, shared transaction/recovery infra) it is close to disqualifying: subtree refcounts are not expressible as "keys are the only metadata." Part (D)/range-read speedup requires a coherent per-version view across journal-order interleaved bsets with NO btree locks held across the scan and bounded kernel latency — the per-key ancestor probe is O(1) via the 128-bit bitmap for the common shallow case (8.3.3), so the probe the proposal attacks is already cheap; the bytes-scanned-from-siblings problem is real but is the price of O(1) create and shared keys, and "fixing" it by de-interleaving versions reintroduces copy-on-create.

**What survives → recommendation.** Drop (B) entirely (refcounted subtree sharing is a philosophical reversion that violates uniformity and re-couples sharing to node lifetime) and treat (D) as out of scope (range-read version-phasing needs a coherent cross-bset view incompatible with the no-IO/no-locks-across-scan and bounded-latency constraints, and the per-key ancestor check is already O(1) for shallow trees via the 128-bit bitmap). What genuinely survives is a narrowed (A): a write-buffer-backed per-snapshot dirty-set is the ONE piece that fits bcachefs's existing patterns (it is exactly how backpointers/LRU/accounting already work — high-churn, coalescing, journalled, crash-consistent), and it attacks a real, doc-acknowledged asymmetry (8.3.5). But even (A) should be justified empirically first: measure whether deleting a low-churn snapshot is actually scan-bound in practice or already fast because the snapshot-aware keyspace iteration is sequential and cache-friendly. If it IS a bottleneck for the thousands-of-short-lived-snapshots CI/backup workload, the right framing is not a new semantic index but extending the EXISTING per-snapshot accounting subsystem (8.3.6 already tracks per-snapshot disk usage / own-vs-cumulative) to also retain a coalesced dirty-key-set, so it shares one recovery/fsck story instead of adding a new invariant. Net deliverable: a benchmark-backed RFC for an accounting-integrated per-snapshot dirty-set that makes deletion track dead data, explicitly NOT touching the in-key snapshot model, NOT adding subtree refcounts, and NOT promising read speedups — and gated on a new online-fsck cross-check being designed up front, not bolted on.

## Read-path latch-freedom & the intent-lock model  — `dubious`

**bcachefs today.** bcachefs already implements Optimistic Lock Coupling (OLC) in all but name. Per 8.7.6, btree nodes use SIX locks (shared/intent/exclusive) with embedded sequence numbers "incremented when taking and releasing exclusive locks (much like seqlocks)" so readers can "aggressively drop locks" and "retake the lock by checking the sequence number rather than redoing the full btree traversal" (8.7.6.3). Because struct btree objects are evicted and reused for different on-disk nodes, "after locking a node the caller must verify it is still the expected node" (8.7.6) — a version/identity check identical to ART's readLockOrRestart/checkOrRestart. The parent-child deadlock (8.7.6.2) is resolved by dropping the read lock on the parent before taking the child intent lock, then "verify the node is still valid and redo the traversal if necessary" — optimistic hand-over-hand with restart. Intent locks let a writer hold position across an entire split while readers keep reading; only the brief final in-memory pointer update takes an exclusive lock (1.2, 8.7.6.1). CoW splits already allocate new physical nodes and update the parent pointer, and the node cache is keyed by physical pointer (8.7.6) — a partial indirection layer. No btree locks are ever held across IO; deadlocks are handled by cycle detection then restart (8.7.6.4).

**Gap claimed.** Two concrete weaknesses remain, both on the read path. (1) OLC restarts: on a SIX exclusive (the brief in-memory pointer update on split/merge/compact, and the rw-bset insert that bumps the node seqlock), a concurrently-validating reader fails its sequence check and must restart — re-walk the affected level. The ART paper flags exactly this: "restarts are particularly undesirable for reads, because they are predominant in many workloads," and a reader "may be restarted repeatedly" in pathological cases. bcachefs's nodes are 256K with multiple bsets, so a restart re-runs the auxiliary-search-tree descent plus per-bset merge — non-trivial work. (2) Reader-writer leaf conflict: the final pointer update and every rw-bset append take an exclusive lock that briefly conflicts with shared readers on that node, and on a hot interior node (e.g. an alloc/LRU/write-buffer btree root, or the root itself which is pinned and traversed by every operation) this serializes readers against the writer for the duration of the exclusive section and forces their revalidation. Neither is a correctness or IO problem — it is a tail-latency / scalability ceiling on hot nodes, which matters precisely because bcachefs's stated goal is "the first general purpose soft realtime filesystem" (1.1) where worst-case revalidation cost on a contended hot node is the thing you most want to bound.

**Proposal (from The ART of Practical Synchronization (Leis, Scheibner, Kemper, Neumann 2016) — OLC vs ROWEX framing; readers never restart under ROWEX, FB+-tree: A Memory-Optimized B+-tree with Latch-Free Update (Chen, Li, Li, Deng 2025) — CAS/optimistic latch-free leaf updates; multiple writers + concurrent readers on one node without blocking, BzTree (Arulraj, Levandoski, Minhas, Larson 2018) — simple latch-free in-place node updates without Bw-tree mapping-table indirection; cautionary measurement of indirection cost).** Adopt ROWEX semantics selectively on the read path — make reads never restart and never check a node-global sequence number — for the leaf/rw-bset path, while keeping OLC + intent locks for structural (split/merge) coordination. Concretely: (a) ROWEX-ize the rw-bset append. Today an insert into the rw bset takes the node exclusive (bumping the seqlock and invalidating all concurrent readers). Instead, write the new bkey into free space at the bset tail and publish it with a single release-store of the bset's count/end offset (the rw-bset already keeps a per-cacheline offset table that tolerates a tail-appended layout), so a reader merging bsets either sees the old end (misses the new key, fine — it was concurrent) or the new end (sees a fully-written key). This is the FB+-tree result — "multiple writers perform updates on one leaf node without blocking other updates nor reads," lookup "executed simultaneously with updates" via CAS on the node rather than a node lock — and it removes the reader-restart on the single most frequent write. Readers stop checking the node seqlock for rw-bset content and instead rely on the monotonically-advancing published end offset, so a concurrent append no longer triggers a restart. (b) For the structural exclusive section (the parent pointer swap on split), keep OLC but borrow BzTree's lesson over the Bw-tree's: do NOT add a Bw-tree-style mapping-table indirection (the ART, BzTree, and FB+-tree papers all measure that its per-node extra dereference "leads to degraded performance on modern CPUs"). bcachefs's physical-pointer-keyed node cache already gives split atomicity without a mapping table, so the pointer swap can stay a short exclusive; only widen ROWEX to the read side so that this swap, like the append, publishes via a release-store the readers acquire-load rather than a seqlock readers must re-validate against. Net: writers still take fine-grained local exclusion only on the node they mutate (the ROWEX invariant — "write locks acquired infrequently by writers and never by readers"), but readers are downgraded from "optimistic + may restart" to "wait-free acquire-load, never restart" on the leaf path.

**Already in bcachefs?** partially: bcachefs already does Optimistic Lock Coupling (SIX locks + seqlock-style sequence numbers + post-lock identity verification + drop-parent-then-revalidate hand-over-hand), which the ART paper presents as the recommended practical sweet spot — so the "optimistic, lock-light, no-mapping-table" half is fully present and arguably already best-practice. What is NOT present is the ROWEX property that READS never restart and never check a version: bcachefs readers still revalidate against the node seqlock and restart on a concurrent exclusive (rw-bset append or pointer swap). It also deliberately does NOT use a Bw-tree mapping-table indirection — and given ART/BzTree/FB+-tree all measure that as a net loss on DRAM, that omission is correct, not a gap. So the honest gap is narrow: convert the hot leaf/rw-bset read path from OLC-with-restart to ROWEX-no-restart; everything else is essentially already there.

**Skeptic's strongest counterargument.** The proposal's central premise — that bcachefs readers "restart" on a concurrent exclusive and that this re-runs the auxiliary-search-tree descent plus per-bset merge — misreads what the seqlock check actually costs in bcachefs, and the doc states the cheaper mechanism explicitly. Per 8.7.6.3, "we can usually retake the lock by checking the sequence number rather than redoing the full btree traversal," and 1.3/1.2 reinforce that "if a node's sequence number hasn't changed, the lock can be reacquired without re-traversing the btree." The sequence-number check is the RELOCK fast-path, not a full re-walk; a reader that loses the seqlock race re-validates one node and, only if it genuinely changed, redoes work scoped to that node — it does not re-descend from the root. So weakness (1) overstates the cost of the very thing it proposes to remove. More fundamentally, the proposal's two halves are asymmetric in difficulty and the hard half is the one bcachefs deliberately closed off. The rw-bset append (half a) is a plausible local optimization, but the structural-swap ROWEX-ization (half b) collides head-on with the documented invariant that makes bcachefs's whole locking discipline cheap: struct btree objects are "never freed during normal operation... which means locks can be dropped and retaken WITHOUT REFERENCE COUNTING" (8.7.6), and unlocked nodes "may be evicted from the cache and reused for different on-disk nodes at any time, so after locking a node the caller must verify it is still the expected node." The proposal itself concedes (tradeoff 2) that an acquire-load reader "must be kept safe against the struct btree being reused" and that "under ROWEX you need an explicit epoch/refcount guard." That is precisely the machinery bcachefs threw out on purpose — the seqlock identity check is the lighter-weight substitute for epoch/refcount reclamation, chosen to keep kernel latency bounded. Re-introducing epoch reclamation or refcounting on the read path to buy "never restart" trades a bounded, cheap re-validation for an unbounded reclamation-deferral problem (epoch stalls) in a soft-realtime kernel context — the opposite of the stated goal. Finally, the uniformity + crash-consistency constraints bite: every reader on every one of 28 btrees would have to adopt the new acquire-load-vs-seqlock protocol and the publish-via-release-store ordering, and the proposal admits (tradeoff 1) the memory-ordering bug it risks is "far subtler to debug than a lock, in unsafe C" — exactly the failure mode bcachefs's restart-based, cycle-detected, idempotent-transaction model was designed to avoid (8.7.6.4: "this approach eliminates deadlocks entirely and keeps worst-case latency bounded").

**What survives → recommendation.** Drop half (b) entirely — the structural-pointer-swap ROWEX-ization is a net loss: it requires re-introducing epoch/refcount reclamation that bcachefs explicitly designed away (8.7.6's "no reference counting" + seqlock identity check), it would create unbounded reclamation-stall latency that contradicts the soft-realtime goal, and it must be done uniformly across all 28 btrees' read paths in unsafe C with subtle memory ordering. The pointer swap is already a "brief in-memory exclusive" by design and the contention it adds is a transient, bounded blip, not a restart. What MIGHT genuinely remain is a much narrower version of half (a): scope a latch-free rw-bset append to the small, empirically-confirmed set of hot single-key/high-churn structures that already have purpose-built front-ends — the key cache and the write buffer — rather than the general btree leaf path. Those layers (a) already coalesce/batch updates so the append pattern is concentrated and uniform, (b) are where "every transaction touches a hot node" is actually true (alloc/LRU/backpointer/accounting), and (c) are already special-cased, so a localized release-store-published append does NOT have to satisfy the all-28-btrees uniformity bar. But this should be gated on profiling that the rw-bset seqlock bump is a measured tail-latency contributor on those nodes under real multi-core load — the doc gives no evidence it is, and the FB+-tree 2.3–3.7x figure is a DRAM-resident OLTP benchmark at 96 threads, not a kernel filesystem metadata workload where IO and journal-commit latency dominate and btree-lock hold times are already sub-microsecond by the "never hold locks across IO" rule. Net: propose only "investigate a release-store-published append for the key-cache/write-buffer hot path, contingent on a profile showing rw-bset exclusion is a real bottleneck," and explicitly recommend AGAINST touching the structural lock protocol.

## Per-node bsets vs cross-level Bε-tree message buffering  — `dubious`

**bcachefs today.** bcachefs batches writes at three distinct layers, none of which is cross-level message buffering in the Bε sense. (1) INTRA-NODE (8.7.4): a btree node is log-structured internally — an initial fully-sorted bset from the last compaction plus journal-order appended bsets, resorted in memory into a geometric progression of typically <=3 bsets; full mergesort happens only on split/rewrite. New keys are appended under a read lock, keeping lock hold times short. This batches writes WITHIN one node but never moves data between levels lazily. (2) JOURNAL (8.6): a WAL records every btree update first; actual node writes are deferred until a large batch accumulates or journal reclaim forces a flush (journal pins tie dirty nodes to the oldest sequence they dirty). This amortizes node-write I/O across many updates but the logical destination of each update is still its final leaf — the journal is redo, not a relocatable buffer. (3) WRITE BUFFER (8.7.10): for a FEW explicitly high-churn btrees (backpointers, LRU, accounting), updates go to a single sorted batching layer that is periodically sorted and flushed, coalescing repeated updates to the same key and amortizing one btree traversal over many updates; the key cache similarly fronts hot single-key btrees (alloc). So bcachefs already has ONE flat (single-level, root-side) message buffer for selected trees, plus per-node log-structuring, plus a journal — but messages are never staged in INTERIOR nodes and flushed DOWN the tree level by level.

**Gap claimed.** The three batching layers leave a gap for the large, deep, insert-heavy snapshot-aware trees (extents, dirents, inodes, backpointers). A genuine Bε-tree divides insert I/O by the per-flush batch size B^(1-e) (intro paper Table 1: insert cost O(log_B N / (e*B^(1-e))) vs B-tree's O(log_B N)), because a key physically moves toward its leaf only after ~B^(1-e) messages have piled up for that subtree, so a single leaf rewrite carries hundreds of updates. bcachefs's per-node log-structuring gives this amortization only for keys that happen to land in the SAME node; a stream of random inserts spread across many leaves still dirties (and eventually rewrites/compacts) many distinct leaf nodes, and the journal only batches the I/O scheduling, not the number of distinct nodes touched. The flat write buffer (8.7.10) is applied to a hand-picked handful of trees and is single-level: it coalesces but does not cascade, so its batching factor is bounded by what fits in one root-side buffer rather than compounding across O(log N) levels. There is no general mechanism for a high-write btree to amortize a random-insert stream across the whole height of the tree the way a Bε internal-node buffer does.

**Proposal (from An Introduction to Bepsilon-trees and Write-Optimization (Bender et al., USENIX ;login: 2015) — interior-node message buffering, B^(1-e) batch flush, insert/point-query cost table, upserts and the 'avoid query-before-update' rule, BetrFS: Write-Optimization in a Kernel File System (Jannen, Yuan, Zhan et al., FAST/TOCS 2015) — in-kernel Bepsilon-tree, mapping VFS microwrites onto buffered messages, kernel-context constraints, Optimizing Every Operation in a Write-Optimized File System (Yuan, Zhan, Jannen et al., FAST 2016) — late-binding journaling (avoid double write of large values), rangecast messages for multi-child propagation (range-delete), zoning).** Add an optional per-btree "interior write-buffer" mode (a Bepsilon flush discipline) that generalizes the existing flat write buffer (8.7.10) into a 2-3 level cascade, reusing bcachefs's own primitives. Concretely: (a) Give interior nodes of opted-in trees (start with backpointers/LRU/accounting, then extents) a bounded message region — implemented as extra appended bsets in the interior node, since interior nodes are already log-structured and already store keyed entries (bch_btree_ptr_v2). Inserts become "messages" appended to the root/interior bset under a read lock (exactly the cheap append path of 8.7.4). (b) Flush a message batch down ONE level only when a node's buffer exceeds a threshold, and ONLY when the whole batch targets few children (the BetrFS "flush straight to the child with the most pending messages" heuristic from the intro paper); a flush is an ordinary multi-key transactional insert into the child, so it rides the existing journal + idempotent-restart + cycle-detection infrastructure unchanged. (c) Cap the cascade depth (e=close to 1, i.e. 2-3 buffered levels, NOT full O(log N)) so query latency stays bounded for the soft-realtime goal. (d) For the read side, adopt UPSERTS / blind writes (intro paper 1.4): expose insert-as-message so the interner-style "read-modify-write" never forces a query-before-update; accounting and LRU updates are already blind increments, ideal upsert candidates. (e) Pair this with BetrFS 0.2 LATE-BINDING JOURNALING for large extent values: log a placeholder/pointer in the journal and bind the final on-disk location at flush time, so a buffered extent value is not written once to the journal and again to the leaf. Lookups merge buffered messages along the root-to-leaf path (commit-order), exactly mirroring how bcachefs already merges across bsets within a node — the in-node eytzinger search tree per bset still applies.

**Already in bcachefs?** partially: bcachefs already has (1) intra-node log-structuring with geometric-progression resort (8.7.4) which is per-node write batching, (2) a journal that defers/batches node writes (8.6), and (3) a genuine — but FLAT, single-level, root-side — write buffer plus key cache for selected high-churn trees (8.7.10) that coalesces updates and amortizes traversal. What remains absent is CROSS-LEVEL message buffering: staging messages in INTERIOR nodes and cascading them DOWN the tree so the batch factor compounds across O(log N) levels and a random-insert stream is amortized across the full height (the defining Bepsilon property of dividing insert cost by B^(1-e)). bcachefs also does not currently expose upserts/blind-write messages or late-binding journaling for large values. So the honest framing is: bcachefs has write-optimization in spirit at three layers, but its cross-node batching is one level deep and tree-selective; a true interior-buffer cascade is the missing piece, and it is deliberately avoided because deeper buffering trades against the kernel's bounded-latency query goal.

**Skeptic's strongest counterargument.** The Bε win the proposal quantifies divides insert cost by the tree HEIGHT (log_B N levels of buffering) — but bcachefs has almost no height to amortize across, so the dominant term the proposal relies on does not exist here. Section 8.7.2 states nodes are 128-256K and trees are "typically 2-3 levels even at petabyte scale." With only 2-3 levels and the leaf already 128-256K, a random-insert stream that the proposal claims "dirties many distinct leaf nodes" is dirtying the SAME small set of huge nodes, and 8.7.4 already names the in-node geometric-progression bset resort as "one of the main ways bcachefs efficiently uses such large btree nodes" — i.e. the per-node log-structuring IS bcachefs's chosen substitute for cross-level buffering, deliberately, because the node is the size at which a B-tree leaf and a Bε leaf-plus-buffer converge. A Bε cascade buys B^(1-ε)/level over ~2 interior levels; the first flush is most of that win, and bcachefs already captures the equivalent of the first flush inside the leaf node itself (append-under-read-lock, coalesce on resort). Worse, the proposal's flush "mutates a child," which violates the load-bearing locking invariant of 8.7.6: normal updates append to a leaf under a READ lock with exclusive taken only for the brief in-memory pointer swap; an interior message that later cascades down forces an intent-lock path on interior nodes and enlarges the per-write node set, directly raising deadlock-cycle-restart frequency (8.7.6.4) and lengthening worst-case latency — the exact bounded-latency / soft-realtime property the doc repeatedly protects. And the read tax is not amortizable: every point query on a buffered tree must merge interior buffers along the root-to-leaf path on the FAST path, whereas today a query touches 2-3 nodes each with <=3 bsets searched via the eytzinger aux tree (8.7.7). The honest reading is that Overstreet already evaluated the design space — he built three batching layers and chose, for exactly the high-churn trees the proposal targets (backpointers/LRU/accounting), a FLAT root-side write buffer (8.7.10) plus key cache, not an interior cascade. That is not an oversight; it is the bounded-latency-preserving subset of Bε that survives the kernel constraints.

**What survives → recommendation.** Drop the interior-node cascade (depth >1) entirely for extents/dirents/inodes — it is precluded by shallow trees + the IO-free read-lock-append leaf model and offers near-zero asymptotic win at 2-3 levels while taxing every point/range query and raising deadlock-restart rate. What genuinely survives the critique is narrow and decoupled from the Bε framing: (1) UPSERTS / blind-write messages (intro paper 1.4) are a real, independently-valuable gap the doc does not mention — accounting and LRU are already blind increments and the interner read-modify-write path forces query-before-update; exposing an upsert message type into the EXISTING flat write buffer (8.7.10), with no interior buffering at all, captures the "avoid query-before-update" win while staying single-level and uniform with whiteout precedent (a buffered KEY_TYPE_deleted is already a coalescable message). This is the only credible, low-risk extraction. (2) LATE-BINDING JOURNALING for large extent VALUES is a legitimately separate optimization (it attacks journal-vs-leaf double-write, not tree height) and is worth a targeted investigation against 8.6's redo journal — but it is orthogonal to message buffering and should be evaluated on its own as a journal change, weighed against the recovery-replay complexity the proposal itself flags (bind step must replay correctly under the 48-pass / blacklisting recovery model). (3) If write amplification on backpointers/accounting is actually measured to be a problem, the in-design lever is to TUNE the existing write buffer (size, flush cadence, wider coalescing window) before adding any new tree class — that stays within the uniformity and online-fsck constraints. Net: pursue upserts-into-write-buffer and a standalone late-binding-journal study; abandon the interior Bε cascade.

## In-node search: eytzinger aux tree vs SIMD/branchless/learned  — `dubious`

**bcachefs today.** bcachefs (sec 8.7.7) has two in-node search paths. (1) Read-only bsets: an eytzinger (BFS-array, heap-layout) auxiliary search tree, one node per cacheline of the original bset, each node storing only ~16 differentiating bits of the first key in that cacheline (computed from the [l,r] bit-range constrained during descent, fixed-size so it falls back to full compare when the differentiating prefix doesn't fit; >99% hit). Descent is prefetch-friendly because children/grandchildren are array-adjacent. The aux tree replaces the cross-cacheline binary search; once it lands on a single cacheline, the code does a LINEAR scan within that cacheline over the actual packed bkeys to find the exact slot. (2) Read-write (currently-inserted-into) bset: a simpler per-cacheline offset table tracking the first key's offset per cacheline, used for a binary search down to a cacheline, then a linear scan within it. Keys are variable-length, packed per-bset via bkey_format (40B bpos -> typically 8-16B). The aux tree was "an enormous improvement... over an order of magnitude faster" than the old plain binary search on large nodes.

**Gap claimed.** The eytzinger tree optimizes only the cross-cacheline navigation (turning it into fixed-width 16-bit prefix compares with cache-friendly prefetch). The final within-cacheline step remains a SCALAR, BRANCHY linear scan over packed variable-length bkeys, plus the >0.5%-of-the-time full-compare fallback on a 160-bit bkey. Per the index-search survey, this is exactly the residual cost modern in-node search techniques target: the per-element compare loop has unpredictable branches (misprediction stalls) and cannot exploit SIMD. bcachefs deliberately does NOT use SIMD/branchless k-ary search (FAST/Schlegel) or BS-tree style data-parallel succ, and the doc gives the reason implicitly: those need fixed-width, contiguous, comparable keys, whereas bkeys are variable-length packed (you can't load N keys into a SIMD register when each is 8-16B and self-delimiting). So a 256KB node with ~16-64 keys per cacheline still pays a branchy scalar tail on every lookup, and on the RW bset the whole descent is branchy binary search with no aux tree at all.

**Proposal (from BS-tree: A gapped data-parallel B-tree (Tsitsigkos et al., 2025) — branchless SIMD succ via cmpge+popcount mask; data-parallel within-node search, Index Search Algorithms for Databases and Modern CPUs (Gross, 2010) — FAST/KCS+10 hierarchical SIMD/cacheline/page blocking; Schlegel k-ary SIMD search; CSB+ context, Making B+-Trees Cache Conscious (Rao & Ross, SIGMOD 2000) — contiguous-key packing, pointer elimination, more comparable keys per cacheline).** Add a fixed-width, SIMD-searchable "key digest" lane PER CACHELINE for read-only bsets, layered under the existing eytzinger tree, to make the within-cacheline tail branchless. Concretely: when the aux search tree is built at sort/compaction time, for each cacheline also materialize a small contiguous array of the SAME ~16-bit (or 32-bit) differentiating-prefix digests that distinguish the keys inside that cacheline (the bit-range machinery already exists in eytzinger1_to_inorder / the [l,r] differentiating-bit computation). Within-cacheline search then becomes a BS-tree/Schlegel-style branchless step: broadcast the search key's digest into a SIMD register, _mm_cmpge against the contiguous digest lane, movemask + popcount to get the rank (BS-tree Snippet 2 succ pattern), and only fall back to a full packed-bkey compare for the (rare) digest-tie slots. This is the FAST [KCS+10] hierarchical-blocking idea (SIMD-width block under the cacheline block under the node) applied at bcachefs's existing cacheline granularity, but using prefix DIGESTS so it works despite variable-length packing — the digest lane is the fixed-width contiguous representation SIMD needs, sitting beside (not replacing) the packed keys. Secondarily, give the RW bset the same per-cacheline digest lane (it currently has only an offset table + plain binary search), making its descent branchless too; the digest lane is cheap to append-maintain since inserts already touch the per-cacheline offset table. Optionally, since interner-style btrees (alloc, backpointers) have dense/near-uniform key distributions, store a tiny per-bset 2-point linear model (Kanva-style: min key, slope) to seed the FIRST cacheline guess instead of starting eytzinger at the root — a learned seek that, on dense keys, lands within a couple of cachelines and shortens the eytzinger descent.

**Already in bcachefs?** partially: bcachefs already does the hard cache-conscious half — the eytzinger aux tree gives prefetch-friendly, fixed-width 16-bit-prefix comparisons for CROSS-cacheline navigation, which is most of what SIMD k-ary search buys you and is why it was 'over an order of magnitude faster' than plain binary search. What remains NOT done: (1) the WITHIN-cacheline step is still a scalar branchy linear scan over packed keys — no branchless/SIMD popcount-rank there; (2) the RW (actively-inserted) bset has only a per-cacheline offset table + plain binary search, no aux tree and no branchless path; (3) no learned/model-based seek. The doc explicitly chose prefix-bit comparisons + linear fallback precisely because keys are variable-length packed, which is the real blocker to naive SIMD — so the honest gap is the digest-lane idea that reconciles SIMD with packing, not 'they forgot SIMD'.

**Skeptic's strongest counterargument.** Section 8.7.7 makes the within-cacheline linear scan a deliberate design choice, not a residual oversight, and the proposal's economics evaporate at the real granularity. Overstreet states the aux-tree key explicitly stores only enough bits to "correctly pivot on the key the current node points to" and to tell it apart from the prior key, with the stated reason: "We're NOT looking for strict equality comparisons here, we're going to follow this up with a linear search anyways." The linear scan is the intended terminal step. Two facts gut the SIMD proposal there: (a) A cacheline is 64B; packed bkeys are 8-16B, so a single cacheline holds only ~4-8 keys, not the "16-64 keys per cacheline" the proposal's limitation section claims (that conflates the whole node with one cacheline). A branchless SIMD cmpge+movemask+popcount over 4-8 digests, with a fixed broadcast/extract setup cost, routinely fails to beat a 4-8 iteration scalar scan that is already fully L1-resident (the cacheline was just loaded for the compare) and whose branch is short and well-predicted. The BS-tree's "up to 7x" is for cap-16 fixed uint64 within-node search over many more elements; it does not transfer to a 4-8 element resident tail. (b) The proposal concedes the aux structure must grow from O(cachelines) to ~O(keys/2) of 2-byte entries. That is a frontal collision with 8.7.7's explicit, repeatedly-stated design objective of making aux nodes "as small as possible" (no original-key pointer, no high bits, no low bits beyond the differentiator, fixed 16-bit with a flag-and-fallback). Doubling-to-octupling aux memory on 256KB nodes worsens the very cache footprint the aux tree exists to shrink, and adds to the geometric-resort build budget on the compaction path. The aux tree was "over an order of magnitude faster" precisely because it cut cache misses on cross-cacheline navigation; the proposal trades that ethos for an in-L1 micro-op shave that may not even be positive, and could be net-negative once the larger aux footprint evicts useful lines. The RW-bset half is the weakest: it adds digest maintenance to the hot insert path under intent/exclusive locks (bounded-latency-sensitive) for a bset the doc deliberately keeps cheap-to-maintain, and the learned-seek half is a hint that on skewed key distributions buys nothing while adding code to the uniform-across-28-btrees fast path.

**What survives → recommendation.** Drop the per-key digest lane and the RW-bset/learned-seek additions; they fight the section's stated "keep aux small, finish with a linear scan, don't do equality at the cacheline" design and likely don't pay off at the real ~4-8-keys-per-cacheline granularity. What genuinely remains, if anything, is narrow and must be settled by measurement, not argument: (1) Make the EXISTING within-cacheline linear scan branchless/autovectorizable WITHOUT any new digest structure — i.e. precompute per-key offsets in the existing per-cacheline metadata so the terminal step is a straight-line predicated compare loop over already-resident packed keys, eliminating the misprediction without growing the aux footprint. This is the only variant that respects the "keep aux small" constraint. (2) Treat it as a benchmark question first: instrument bch2_btree_node_iter on the hottest point-lookup btrees (inodes, alloc via key-cache) and confirm the terminal scan is actually a measurable fraction of lookup cycles after the eytzinger descent — if the >99% prefix-hit path already resolves most lookups to one or two candidate slots, there is essentially nothing to vectorize. (3) Validate this in the workspace's own RadixBimap/interner experiment (dense small-integer ids) entirely in userspace before ever proposing kernel C — that is where dense keys make a model/branchless tail plausible, and it carries zero crash-consistency, online-fsck, or NO-IO-under-locks risk. Anything kernel-bound should ship only with a per-arch scalar fallback and a same-format-invalidation check on the seqlock re-traversal path, and only after (2) shows a real win.

## Cache-obliviousness vs hand-tuned 128-256K nodes  — `dubious`

**bcachefs today.** bcachefs deliberately sits at the cache-AWARE end of the spectrum on two distinct levels. (1) Macro level: section 1.1 says btree nodes are "very large" and 8.7.5/8.7.7 fix them at 128-256K (typically 256K), explicitly chosen so the filesystem can run its own node cache and tune reclaim "independently of the page cache" rather than being limited to 4K page-cache metadata like other COW FSes. The 256K node is the I/O-transfer unit (contiguous on-disk region, sequential write of appended bsets). (2) Micro level: the in-node auxiliary search tree (8.7.7) is an eytzinger / BFS-array layout with EXACTLY one node per cacheline, each storing only ~16 differentiating bits of the first key in that cacheline (computed offset via eytzinger1_to_inorder(), fallback to full bkey compare). The doc itself names the benefit as cache-awareness: walking it "corresponds roughly to a binary search... but with much friendlier memory access patterns, since at every iteration the children of the current node are adjacent in memory (and all the grandchildren, and all the great grandchildren)" — i.e. it is explicitly cacheline-granular and prefetch-friendly. So bcachefs already harvests most of the locality win, but it does so by hand-tuning B at two scales (256K for the storage/DRAM transfer, 64B cacheline for the aux-tree node).

**Gap claimed.** The cache-awareness is single-granularity and hardcoded. The aux tree is optimized for exactly one transfer size (the 64B cacheline / one aux node per cacheline). It is NOT a recursive van Emde Boas (vEB) layout, so it is only tuned for the L1-line transfer; it does not simultaneously optimize transfers at the L2 page, the 4K TLB page, the huge-page, or the 256K btree-node-to-DRAM boundary. Concretely: when you binary-search the eytzinger array of a 256K node, the FIRST ~log2(node_size/64) levels each touch a fresh, far-apart cacheline (the array root and its near descendants are scattered across the node), so the upper half of the descent is a sequence of near-random cacheline misses with no spatial locality between successive probes; only the last few levels (once the working subtree fits in a line) get the adjacency benefit the doc describes. A vEB layout would block-nest those upper levels too, cutting misses for the cold portion of the descent at every memory level at once. Separately, the 256K node size is a global constant: it is good for spinning disks / petabyte scale (1.1 motivates it that way) but for an in-memory-resident, point-lookup-heavy btree (e.g. the alloc btree fronted by the key cache, or an interner-style workload) 256K nodes mean more wasted DRAM transfer and L2/L3 pollution per lookup than the data actually needed. bcachefs has to pick one B for all 28 btrees.

**Proposal (from Cache-Oblivious B-Trees (Bender, Demaine, Farach-Colton, SICOMP 2005) — the van Emde Boas static layout giving Theta(1+log_{B+1}N) transfers at all hierarchy levels with B unknown; their Section 2.1 vEB layout is exactly the drop-in for the aux tree, Cache-Oblivious Streaming B-trees / COLA (Bender et al., SPAA 2007) — confirms the CO model's premise that B is unknown and that a single layout can be optimal across levels; its write-optimization angle is mostly already captured by bcachefs's log-structured node + geometric bset progression, so it is cited as a boundary (why NOT to chase it here), Making B+-Trees Cache Conscious in Main Memory / CSB+ (Rao, Ross, SIGMOD 2000) — the cache-AWARE baseline bcachefs already resembles (pointer elimination via contiguous children + arithmetic offsets, packed keys per line); cited to argue bcachefs has already captured the CSB+ win and the remaining headroom is the multi-level/oblivious dimension, not more single-level cache-conscious packing).** Replace the single-cacheline-granular eytzinger aux tree with a recursive van Emde Boas (vEB) layout of the same per-cacheline summary nodes, keeping everything else (the ~16-bit differentiating-key trick, fallback-to-full-compare, fixed-size nodes flagged if they don't fit). Concretely: take the existing array of aux nodes (one per cacheline of the bset), and instead of laying them out in BFS/heap order, lay them out in vEB order (recursively split the tree of height h at height h/2, store the top recursive subtree contiguously, then each bottom subtree contiguously). Bender-Demaine-Farach-Colton show this single static layout achieves Theta(1 + log_{B+1} N) transfers at EVERY level of the hierarchy simultaneously with B unknown — so the same aux array would be near-optimal for the L1 line, the L2/L3 line, AND the 4K/huge TLB page, removing the hand-tuning of "one node per cacheline" as the only optimized granularity. The construction is still a static, build-once array recomputed at bset resort/compaction (the aux tree is already rebuilt then), so it adds zero per-update cost; the index math just changes from eytzinger1_to_inorder() to a vEB position function (precomputable into a small per-height lookup table since node geometry is fixed). As a complementary, lower-risk second step aimed at the macro granularity: rather than making the 256K node size adaptive (which would break the uniformity constraint), keep the on-disk node at 256K but give the IN-MEMORY first/sorted bset of cache-resident, point-lookup-dominated btrees a vEB-laid-out aux tree so the resident lookup cost is hierarchy-optimal without changing the on-disk format or the transaction/recovery infra.

**Already in bcachefs?** partially: bcachefs is already strongly cache-AWARE and has captured most of the practical locality win — the aux tree is an eytzinger/BFS array with one node per cacheline, ~16 differentiating bits, and explicit prefetch-friendly child-adjacency (8.7.7), and the node layout already does CSB+-style pointer elimination + key packing. What is NOT present is cache-OBLIVIOUSNESS: the layout is hand-tuned for a single granularity (the 64B cacheline) and a single global node size (256K), not a recursive van Emde Boas layout optimal across L1/L2/L3/TLB/DRAM simultaneously. So the missed opportunity is narrow and specific (multi-level / B-unknown optimality of the in-node search array), not "they don't do cache-conscious layout."

**Skeptic's strongest counterargument.** The proposal's headline benefit evaporates under the actual numbers. A 256K node has at most 4096 cachelines, so the aux tree has at most ~4096 nodes; with the "as small as possible" entries the doc describes (~2-4B: 16 differentiating bits + offset), the *entire* aux tree is ~8-16KB and spans only 4 TLB pages at the very most (one huge page covers it; the first/dominant bset is the only large one, later bsets are 8-16K total). So the claimed "fewer distinct 4K/TLB pages per lookup" win has almost no surface area — there are only ~4 pages to begin with — and the L2/L3 win is thin because an 8-16KB array is L2-resident for exactly the hot/alloc/interner btrees the proposal targets. The proposal itself concedes (tradeoff 1) vEB "buys little" once the node is L2-resident; the size calc shows that's the common case, not the corner. Worse, eytzinger is NOT "random misses on the upper half": BFS order packs the TOP of the tree most contiguously (root+2 children+4 grandchildren = first 7 entries = 1-2 cachelines), so the cheap-to-fix levels are already cheap. Three further problems: (a) the doc explicitly says the aux tree is "not full (not size 2^n-1)" — vEB's split-at-h/2 recursion is clean only for complete trees; a ragged tree turns the position function from "mostly shifts and bit ops" (the doc's stated reason for choosing eytzinger) into a lookup table that competes for cache with the very array it indexes; (b) the empirical literature (Khuong & Morin, Array Layouts for Comparison-Based Searching) found eytzinger+prefetch matches or beats vEB on real hardware at exactly these sizes because vEB's address arithmetic and weaker prefetch stride cost more than the marginal locality saves — so Overstreet's choice is the known-good answer, not an oversight; (c) the "complementary second step" (give only resident point-lookup btrees a vEB aux tree) directly VIOLATES the everything-is-one-btree uniformity constraint by forking the hottest file (bset.c) into two lookup paths — the exact per-btree special-casing the design exists to forbid.

**What survives → recommendation.** Drop the vEB layout swap and the per-btree second step — both are net-negative or constraint-violating. What genuinely remains is narrower and is a measurement question, not a redesign: (1) If anything, the actionable lever is software prefetch tuning of the EXISTING eytzinger descent (prefetch the eytzinger children/grandchildren block one or two levels ahead), which the doc's "children/grandchildren adjacent" remark already sets up and which keeps the cheap shift/mask index math — this is the part of the literature that actually moved the needle, not the layout change. (2) The only defensible vEB experiment is purely empirical and gated: build a vEB variant of the aux tree behind a benchmark, measure cache-cold lookup latency on a large (256K) cold leaf for a comparison-heavy btree (extents/dirents, NOT the alloc/key-cached ones which are too small to benefit), and accept it ONLY if it wins uniformly across all 28 btrees (preserving uniformity) AND survives the non-full-tree index-math cost. Given the size analysis and the Khuong-Morin result, the prior on that experiment paying off is low; frame it as a falsification test, not an improvement. (3) The macro 256K-node-size point the proposal correctly declines to pursue should stay declined.
